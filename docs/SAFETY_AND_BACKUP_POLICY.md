# Finera Safety and Backup Policy

## Հիմնական սկզբունք

Finera-ն հաշվապահական ծրագիր է, ուստի real/production փուլում ոչ մի վտանգավոր փոփոխություն չպետք է արվի առանց backup-ի, ստուգման և մարդու հաստատման։

## Ոսկե կանոններ

1. Ոչ մի վտանգավոր փոփոխություն առանց backup-ի։
2. Ոչ մի backup առանց restore test-ի։
3. Ոչ մի real accounting փոփոխություն առանց մարդու հաստատման։
4. Ոչ մի production deploy առանց նախնական test/sandbox ստուգման։
5. Ոչ մի database migration առանց նախնական backup-ի և rollback պլանի։
6. Ոչ մի տվյալների ջնջում առանց առանձին հաստատման։
7. AI-ն կարող է առաջարկել, պատրաստել, ստուգել, բայց real accounting գործողությունը վերջնական հաստատում է մարդը։

## Միջավայրեր

Finera-ն պետք է ունենա առնվազն երեք միջավայր.

### Development

Այստեղ կառուցում ենք նոր էջեր, ձևեր, մենյուներ, տեխնիկական փոփոխություններ։

### Sandbox / Test

Այստեղ փորձարկում ենք ծրագրի աշխատանքը test կամ real-like տվյալներով։  
Այս միջավայրը չպետք է օգտագործվի որպես պաշտոնական հաշվապահական հաշվառում։

### Production

Այստեղ պահվում և մշակվում են իրական աշխատանքային տվյալները։  
Production փոփոխությունները պետք է արվեն միայն ստուգումից, backup-ից և հաստատումից հետո։

## Backup քաղաքականություն

Production փուլում պետք է լինեն հետևյալ backup-ները.

### Daily backup

Ամենօրյա ավտոմատ պահուստային պատճեն։

### Before-change backup

Ամեն մեծ փոփոխությունից, migration-ից կամ deploy-ից առաջ առանձին backup։

### Restore test

Պարբերաբար պետք է ստուգել, որ backup-ից իրականում հնարավոր է վերականգնել համակարգը։

Backup-ը արժեք ունի միայն այն դեպքում, եթե restore-ը փորձարկված է։

## Master DB և Tenant DB backup

Finera-ի ապագա ճարտարապետության մեջ նախատեսվում է.

- Master DB
- առանձին Tenant DB յուրաքանչյուր սպասարկվող գործընկերոջ համար

Դրա համար backup-ները նույնպես պետք է բաժանվեն.

### Master DB backup

Պահում է՝

- օգտատերեր
- դերեր
- թույլտվություններ
- սպասարկվող գործընկերների registry
- համակարգային կարգավորումներ
- ընդհանուր reference/master data

### Tenant DB backup

Պահում է կոնկրետ սպասարկվող գործընկերոջ հաշվապահական տվյալները։

Եթե մեկ tenant DB-ում խնդիր առաջանա, դա չպետք է վնասի ամբողջ համակարգը կամ մյուս գործընկերների տվյալները։

## Փոփոխությունների վերահսկում

Production փոփոխության ճիշտ ճանապարհը.

1. Idea / requirement
2. Development
3. Sandbox test
4. Human review
5. Backup
6. Migration / deploy
7. Post-deploy check
8. Rollback, եթե խնդիր կա

## Հաշվապահական logic-ի փոփոխություններ

Հաշվապահական կամ հարկային logic-ի փոփոխությունները համարվում են բարձր ռիսկային։

Օրինակ՝

- ձևակերպումների կանոններ
- հարկային հաշվարկներ
- փաստաթղթի համարակալման կանոններ
- փակված ամսվա փոփոխություն
- հաշվետվության բանաձևեր
- posting logic

Այս փոփոխությունները պետք է արվեն միայն test օրինակներով ստուգելուց և մարդու հաստատումից հետո։

## Audit log

Ապագա production համակարգում պետք է պահպանվի audit log.

Audit log-ը պետք է ցույց տա՝

- ով է արել գործողությունը
- երբ է արել
- ինչ է փոխել
- որ սպասարկվող գործընկերոջ տվյալների վրա է ազդել
- ինչ էր նախկին արժեքը
- ինչ դարձավ նոր արժեքը

## Կարճ կանոն

Finera-ում real data-ի հետ աշխատելիս հիմնական կանոնը սա է.

Ոչ մի վտանգավոր փոփոխություն առանց backup-ի։
Ոչ մի backup առանց restore test-ի։
Ոչ մի real accounting փոփոխություն առանց մարդու հաստատման։
